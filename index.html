Simulation model catalog vocabulary

MoCAT is a vocabulary for describing simulation models. This document lists the requirements, defines the schema and provides examples.

A simulation model is a tool to make a digital representation of a real-world phenomenon in nature, within a population, in a transport network, etc. We use simulation models to better understand observations of the past or to study the impact of changes in the future.

A common vocabulary for describing simulation models on the Web enhances its discoverability and automation potential.

Requirements

Discover simulation models

Users can discover simulation models by searching for and comparing simulation models using various descriptive metadata fields:

  • title
  • description
  • keywords
  • domain
  • legal constraints
  • usage constraints
  • ownership
  • licensing
  • quality assessments such as those in the [[VOCAB-DQV]]
  • validation reports
  • Compliance with internal or regulatory standards. For example, Departement Omgeving speaks of "MER compliance".
  • source code repository URL
  • releases URL
  • collaboration
  • data services
  • Algorithm
  • Invocation mode: whether results of a simulation run are presented synchronously or asynchronously to a user.
  • Service endpoints: the service endpoints where agents are available to execute a simulation run.

A simulation model is the whole of method, algorithm, implemented in software, documentation, etc. A simulation model can be used as software or as a service.

Test interoperability of input datasets

Users must be able to assess via automated reasoning which input datasets are compatible with a simulation model based on the data specification.

Start a simulation run via a universal interface

It must be possible to run simulation models via a universal Application Programming Interface (API). The input and output parameters of the API methods are based on a common metamodel. A user can configure and execute a simulation run, inspect the results (output datasets) of previous simulation runs, etc.

Metadata represents the algorithm of a simulation model as a number of steps and their execution logic. This makes the internal functioning of a simulation model accessible in a standardized way. Ontologies such as [[P-PLAN]] can be a basis for this.

Make a simulation run reproducible

Metadata captures version history, parameter sets, data provenance, and software dependencies so that simulation results can be reproduced, audited, or certified, even when original creators are no longer involved. [[PROV-O]]

Vocabulary specification

The MoCAT vocabulary makes use of the [[VOCAB-DCAT-3]], [[PROV-O]], and [[P-PLAN]] specifications. The MoCAT vocabulary specification is available at mocat-ontology.ttl.

Class: Simulation Model (prov:Plan)

A simulation model is a tool to make a digital representation of a real-world phenomenon.

A simulation model may have a basis in physics, math and/or statistics. In addition to this theoretical foundation, the simulation model is usually implemented in source code and software. We see the simulation model as the whole: the mathematical or physical model, the algorithm, and the software that makes it executable. 

omg-simulationmodel:traffic-scout
    a                   prov:Plan ;
    dcterms:type        mocat:stimulation_model ;
    dcterms:title       "Traffic Scout"@en ;
    dcterms:description """Traffic Scout combines an interactive map and a simulation tool to provide insight into local
traffic flows based on, among other things, local traffic sensors. The technology is based on Telraam data,
traffic indicators, geodata, and OpenStreetMap. The TS model supports model calculations (including the addition
of network changes) for near-synchronous operation. The TS tool focuses on applications at the meso-level
(municipality, district, neighborhood)."""@en ;
    dcterms:publisher   ex:TML ;
    dcterms:creator     ex:TML, ex:CEGEKA ;
    dcterms:issued      "2020-03-15"^^xsd:date ;
    dcterms:modified    "2020-03-15"^^xsd:date ;
    dcat:keyword        "traffic"@en, "simulation"@en ;
    dcat:contactPoint   ex:TrafficScoutContact ;
    dcat:distribution   omg-distribution:traffic-scout-jar-dist ;
    mocat:input         omg-schema:TransportNetworkSchema .

Property: inputSchema

The input schema of a [=simulation model=] is a data specification that describes how an input dataset of a [=simulation run=] is structured.

Property: outputSchema

The output schema of a [=simulation model=] is a data specification that describes how an output dataset of a [=simulation run=] is structured.

Class: Data Specification (p-plan:Variable)

A data specification describes how a dataset must be structured.

Each input and output of a simulation model can have its own data specification. The input or output data is not necessarily structured as RDF, but the schema must be represented as RDF so that it can be used for automated reasoning about the interoperability of input datasets. Hence, we require an RDF description of the data structure in the form of [[SHACL]] NodeShapes and PropertyShapes. 

omg-schema:TransportNetworkSchema
    a              p-plan:Variable ;
    rdf:type       <http://purl.org/adms/assettype/Schema> ;
    rdfs:label     "Transport Network schema"@en ;
    sh:description """A transport network consists of vertices and edges.""" ;
    sh:node        omg-shacl:TransportVertexShape ;
    sh:node        omg-shacl:TransportEdgeShape .

omg-shacl:TransportVertexShape
    a              sh:NodeShape ;
    sh:targetClass ex:TransportVertex ;
    sh:property    [ sh:path     ex:geom ;
                     sh:datatype geo:wktLiteral ;
                     sh:minCount 1 ;
                     sh:maxCount 1 ;
                     sh:message  "A transport vertex must have exactly one (point) geometry (geom)." ; ] ;
    sh:sparql      [ sh:message "Each transport vertex must be connected to at least one transport Edge." ;
                     sh:select  """
            SELECT ?this
            WHERE {
                FILTER NOT EXISTS { ?edge ex:source ?this . }
                FILTER NOT EXISTS { ?edge ex:target ?this . }
            }
        """ ; ] .

omg-shacl:TransportEdgeShape
    a              sh:NodeShape ;
    sh:targetClass ex:TransportEdge ;
    sh:property    [ sh:path     ex:length ;
                     sh:datatype xsd:double ;
                     sh:minCount 1 ;
                     sh:maxCount 1 ;
                     sh:message  "A transport edge must have exactly one length property." ; ] ;
    sh:property    [ sh:path     ex:source ;
                     sh:class    ex:TransportNode ;
                     sh:minCount 1 ;
                     sh:maxCount 1 ;
                     sh:message  "A transport edge must have exactly one source vertex." ; ] ;
    sh:property    [ sh:path     ex:sink ;
                     sh:class    ex:TransportNode ;
                     sh:minCount 1 ;
                     sh:maxCount 1 ;
                     sh:message  "A transport edge must have exactly one sink vertex." ; ] ;
    sh:property    [ sh:path     ex:geom ;
                     sh:datatype geo:wktLiteral ;
                     sh:minCount 1 ;
                     sh:maxCount 1 ;
                     sh:message  "A transport edge must have exactly one (linestring) geometry (geom)." ; ] .

Class: Simulation Run (prov:Activity)

A simulation run is a single, complete execution of a [=simulation model=]. It has a run configuration that defines the starting conditions (parameters and random seeds), stopping point, and input datasets.

Each input and output of a simulation model can have its own [=data specification=]. The data is not necessarily structured as RDF. A semantic asset [[VOCAB-ADMS]] can be used to describe the data structure. In the case of RDF data, [[SHACL]] must be used. 

omg-simulationrun:abc123
    a                      prov:Activity ;
    dcterms:type           ex:SimulationRun ;
    rdfs:label             "Simulation Run abc123"@en ;
    prov:startedAtTime     "2026-06-01T00:00:00Z"^^xsd:dateTime ;
    prov:endedAtTime       "2026-06-01T00:12:59Z"^^xsd:dateTime ;
    prov:wasAssociatedWith ex:TSSoftwareAgent ;
    prov:wasDerivedFrom    omg-simulationmodel:traffic-scout ;
    prov:used              omg-dataset:osm_tn_dataset ;
    prov:used              ex:run_configuration .

Property: used (prov:used)

The used property of a [=simulation run=] is a reference to one or more input datasets that were used. The input datasets must adhere to the [=input schemas=] of the [=simulation model=].

Property: was derived from (prov:wasDerivedFrom)

The was derived from property of a [=simulation run=] relates to the [=simulation model=] of which the simulation run is an execution.

Class: Dataset (dcat:Dataset)

A dataset is defined in [[DCAT]] as a collection of data, published or curated by a single agent, and available for access or download in one or more representations.

omg-dataset:osm_tn_dataset
    a                   dcat:Dataset ;
    dcterms:conformsTo  omg-schema:TransportNetworkSchema ;
    dcterms:title       "OpenStreetMap transport network"@en ;
    dcterms:description """This dataset contains a routable transport network
                       that was derived from OpenStreetMap data."""@en .

Property: was generated by (prov:wasGeneratedBy)

The was generated by property of a [=dataset=] is a reference to the [=simulation run=] that generated the output dataset.

. This output dataset must adhere to the [=output schema=] of the [=simulation model=].

Property: conforms to (dcterms:conformsTo)

The conforms to property of a [=dataset=] is a reference to the [=data specification=] the dataset adheres to.

Implementation report

Metadata creation

Departement Omgeving wants to have an overview of all simulation models that are used in for public service delivery. Simulation models are mainly used for environmental impact assessment.

We will incorporate the creation of metadata for simulation models by embedding this in the software development lifecycle of the simulation model. A release of a new version of the simulation model by the software build engine will trigger the creation of a new metadata record that will be included in a Triplestore. The triple store is schema-free and enables various use cases. Use case 12 in WP4 of the [[X-CITE-P-PLAN]] citiverse project.

Metadata will be available on the Linked Data service of Departement Omgeving at https://data.omgeving.vlaanderen.be/sparql. The metadata can there be explored from a plain SPARQL interface.

Metadata publication and harvesting

We will publish a limited subset of the metadata as ISO19115 metadata records using the Geonetwork catalog. This catalog provides a configurable facetted search functionality via ElasticSearch. The catalog has standard APIs that are used to harvest metadata records. One downside of the use of ISO19115 metadata records is that not all metadata fields in MoCAT can be mapped to this standard.
