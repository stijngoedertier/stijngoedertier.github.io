Simulation model catalog vocabulary

MoCAT is a vocabulary for describing simulation models. This document lists the requirements, defines the schema and provides examples.

A simulation model is a tool to make a digital representation of a real-world phenomenon in nature, within a population, in a transport network, etc. We can use simulation models to better understand observations of the past or to study the impact of changes in the future.

A common vocabulary for describing simulation models on the Web enhances its discoverability and automation potential.

Requirements

The following table summarizes the use cases for a simulation model catalog.
Summary Description Priority
1. Discover simulation models Users must be able to find, compare, and obtain relevant simulation models. Must
2. Test interoperability of input data structure Users must be able to assess via automated reasoning which input datasets are compatible with a simulation model based on the data specification. Could
3. Trigger a simulation run via a universal interface It must be possible to run simulation models via a universal Application Programming Interface (API). It input and output parameters of the API methods are based on the MoCat metamodel to build and execute a simulation run, inspect the run configuration and results (output datasets) of previous simulation runs, etc. Could
4. Make a simulation run reproducible Metadata captures version history, parameter sets, data provenance, and software dependencies so that simulation results can be reproduced, audited, or certified, even when original creators are no longer involved. [[PROV-O]] Won't
5. Make the execution logic of a simulation model accessible Metadata represents the algorithm of a simulation model as a number of steps and their execution logic. This makes the internal functioning of a simulation model accessible in a standardized way. Ontologies such as [[P-PLAN]] can be a basis for this. Won't

Discover simulation models

Users can discover simulation models by searching and comparing simulation models on various descriptive metadata fields:

  • title
  • description
  • purpose
  • origin
  • keywords
  • domain
  • legal constraints
  • usage constraints
  • ownership
  • licensing
  • quality assessments such as those in the [[VOCAB-DQV]]
  • validation reports
  • Compliance with internal or regulatory standards. For example, Departement Omgeving speaks of being "MER compliant".
  • source code repository URL
  • releases URL
  • collaboration
  • data services
  • Algorithm
  • Invocation mode: whether results of a simulation run are presented synchronously or asynchronously to a user.
  • Service endpoints: the service endpoints where agents are available to execute a simulation run.

A simulation model is the whole of method, algorithm, implemented in software, documentation, etc. A simulation model can be used as software or as a service.

Vocabulary specification

The MoCAT vocabulary makes use of the [[VOCAB-DCAT-3]], [[PROV-O]], and [[P-PLAN]] specifications. The MoCAT vocabulary specification is available at

Simulation Model

A simulation model is a tool to make a digital representation of a real-world phenomenon.

This digital representation has a theoretical and mathematical basis. The simulation model is usually implemented in source code and software.

Data Specification

A data specification describes how data is structured.

Each input and output of a simulation model can have its own data specification. The data is not necessarily structured as RDF. A semantic asset [[VOCAB-ADMS]] can be used to describe the data structure. In the case of RDF data, [[SHACL]] can be used.

Implementation report

Metadata creation

Departement Omgeving wants to have an overview of all simulation models that are used in its public services. This includes the MER.

We will incorporate the creation of metadata for simulation models by embedding this in the software development lifecycle of the simulation model. A release of a new version of the simulation model by the software build engine will trigger the creation of a new metadata record that will be included in a Triplestore. The triple store is schema-free and enables various use cases. Use case 12 in WP4 of the [x-cite citiverse](https://xcitecitiverse.eu/) project.

Metadata publication and harvesting

We will publish a limited subset of the metadata as ISO19115 metadata records using the Geonetwork catalog. This catalog provides a configurable facetted search functionality via ElasticSearch. The catalog has standard APIs that are used to harvest metadata records. One downside of the use of ISO19115 metadata records is that not all metadata fields in MoCAT can be mapped to this standard.
